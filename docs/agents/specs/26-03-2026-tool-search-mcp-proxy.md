## Problem Summary

Current MCP architecture forces models to load full tool schemas (8,000–20,000+ tokens) from all backend servers on every request, even when the model uses only a small subset. This creates unnecessary context window overhead and reduces efficiency.

**Goal**: Build an MCP proxy server that wraps multiple MCP backends through 4 lightweight tools (`search_tools`, `get_tool_schema`, `call_tool`, `list_servers`), exposing a searchable tool catalog instead of raw schemas.

### Phase 3: The 4 Proxy Tools (Est: 4 days)

- [ ] **Tool 1: `search_tools`**
  - Input: `query` (string), `max_results` (int, default=10)
  - Load `catalog.json`, iterate all tools
  - Use RapidFuzz token_set_ratio() for name + description matching
  - Rank by score, return top N as `[{ server, name, description, score }, ...]`
  - Test: exact match, partial/fuzzy match, collision resolution, empty result

- [ ] **Tool 2: `get_tool_schema`**
  - Input: `server` (string), `tool_name` (string)
  - Look up tool in catalog, return full `inputSchema` (JSONSchema)
  - Validate tool exists, return error if not found
  - Return metadata: `{ server, tool_name, description, inputSchema, updated_at }`

- [ ] **Tool 3: `call_tool`**
  - Input: `server` (string), `tool_name` (string), `arguments` (object)
  - Get connection from manager, execute tool on backend
  - Validate arguments against schema (raise 400 on validation failure)
  - Forward full response from backend, wrap errors
  - Return: `{ success: bool, result?: any, error?: string, elapsed_ms: number }`
  - Test: success, validation error, tool not found, backend timeout, connection failure

- [ ] **Tool 4: `list_servers`**
  - Input: none
  - Enumerate all backends from `backends.json`
  - Return: `[ { id, name, type, status: 'ready'|'initializing'|'error', tool_count, last_cataloged_at }, ... ]`
  - Status indicators: cached?✓, connection alive?✓, errors?

### Risk Assessment

**High-Risk Items**:

- [ ] **Async resource cleanup**: Multiple spawned processes + open connections. Risk: zombie processes, file descriptor leaks. Mitigation: use `finally` blocks, async context managers, comprehensive cleanup tests.
- [ ] **Schema fidelity**: Forwarding JSONSchema objects without mutation. Risk: invalid schema breaks model validation downstream. Mitigation: parse and re-serialize to validate, add schema validation tests.
- [ ] **Connection state races**: Concurrent tool calls to same backend + 5-min timeout. Risk: destroy connection while in-use. Mitigation: add reference counting or lock-based serialization to `ConnectionManager`.

**Medium-Risk Items**:

- [ ] **Tool name collisions**: Namespacing strategy (`{server}/{tool}`) requires user education. Mitigation: document clearly, provide examples in README.
- [ ] **Malformed tool schemas**: Backend returns invalid JSONSchema; catalog build fails. Mitigation: validate schema during catalog build, log + skip bad tools, don't crash.
- [ ] **HTTP timeout tuning**: 30s default may be too strict or too loose. Mitigation: make configurable per-backend, add monitoring/metrics.

**Performance Assumptions**:

- Catalog.json fits in memory (< 10MB for 50 servers × 40 tools)
- Fuzzy search on 2,000 tools completes in < 100ms (RapidFuzz is optimized)
- Stdio process startup < 5s (subprocess overhead acceptable)

## Implementation Notes

### Configuration Strategy

Backends are defined in `backends.json` (relative to proxy working directory), using a structure similar to `config.json`:

```json
{
  "servers": {
    "context7": {
      "type": "http",
      "url": "https://mcp.context7.com/mcp",
      "timeout_seconds": 30
    },
    "github": {
      "type": "stdio",
      "command": "npx",
      "args": [
        "@modelcontextprotocol/server-github",
        "--token",
        "${GITHUB_TOKEN}"
      ],
      "env": {
        "GITHUB_TOKEN": "${env:GITHUB_TOKEN}"
      }
    },
    "local_tool": {
      "type": "stdio",
      "command": "python",
      "args": ["my_server.py"]
    }
  }
}
```

### Catalog Refresh Strategy

- **Catalog is built once at startup** (or manually via `catalog-builder` CLI)
- For long-running proxies, catalog can be refreshed via:
  - Manual trigger: `python -m src.catalog.builder --config backends.json`
  - Scheduled job: APScheduler running refresh() every 1 hour
  - Not required for basic MVP

### Backwards Compatibility

This proxy is a **new entrypoint** and does not modify existing MCP client/server behavior. It's opt-in: models/clients can call the proxy or call backends directly.
